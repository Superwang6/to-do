# 开发流水线

本流程用于把需求从想法推进到可验证完成。目标是减少返工、漏测和前后端契约不一致。

## 流程总览

```text
需求输入
  ↓
需求澄清与范围确认
  ↓
验收标准
  ↓
影响范围与技术设计
  ↓
实现计划
  ↓
开发执行
  ↓
自动化测试 / 手动验证
  ↓
回归检查
  ↓
完成总结
```

## 阶段 1：需求澄清

### 入口条件

- 有一个明确的问题、功能想法或改动目标。

### 输出内容

- 用户故事
- 功能范围
- 不做的内容
- 入口与交互路径
- 成功、失败、空状态
- 是否涉及 API、数据库、前端页面、布局或配置

### 出口条件

- 需求目标清楚。
- 关键不确定点已确认，或已记录合理假设。

## 阶段 2：验收标准

### 入口条件

- 需求目标和范围已明确。

### 输出内容

使用可验证的 checklist：

```md
## 验收标准
- [ ] 在 <条件> 下，用户执行 <操作>，系统应 <结果>
- [ ] 当 <异常情况> 时，系统应 <错误处理>
- [ ] 原有 <关键流程> 不受影响
```

### 出口条件

- 每条验收标准都可以通过测试或手动操作判断通过/失败。

## 阶段 3：设计与影响范围

### 入口条件

- 验收标准已完成。

### 输出内容

- 后端改动点：schema、model、repository、service、router、tests
- 前端改动点：api、页面、组件、状态、样式
- API 契约：method、path、request、response
- 数据结构和兼容性影响
- 测试策略

### 出口条件

- 已明确要改哪些文件或模块。
- 已明确如何验证。

## 阶段 4：实现计划

### 入口条件

- 设计方案已明确。

### 输出内容

把工作拆成可执行步骤：

```md
## 实现计划
1. ...
   - 验证：...
2. ...
   - 验证：...
```

### 出口条件

- 每一步都有对应验证方式。
- 非平凡修改已得到用户确认。

## 阶段 5：开发执行

### 后端顺序

```text
schema → model → repository → service → api/router → tests
```

约束：

- `api` 只处理请求/响应，不写业务逻辑。
- `service` 处理业务规则和事务编排。
- `repository` 封装数据库 CRUD，不引用 HTTP 层。
- 模型不使用外键；关系必须显式 `primaryjoin`。

### 前端顺序

```text
api 封装 → 页面状态 → UI/交互 → 样式布局 → 联调
```

约束：

- API 路径、方法、字段必须和后端一致。
- uni-app 页面遵守布局规则，避免页面级滚动条和横向溢出。

## 阶段 6：验证与测试

根据改动范围执行：

- 后端：`pytest` 或 `uv run pytest`
- 前端：项目已有 build/lint/test 命令
- API：校验 method、path、request、response
- UI：手测加载、空状态、错误状态、成功状态、布局溢出
- 回归：创建、查看、修改、删除等主流程

测试结果必须记录为：

```md
## 验证结果
- 命令：...
- 结果：通过 / 失败 / 未运行
- 说明：...
```

## 阶段 7：完成总结

最终输出：

```md
## 完成情况
- ...

## 修改文件
- ...

## 验证结果
- ...

## 验收标准结果
- [x] ...
- [ ] ...（原因）

## 风险与下一步
- ...
```

## 失败回退规则

| 失败点 | 回退位置 |
|---|---|
| 需求不清 | 需求澄清 |
| 验收标准不可测 | 验收标准 |
| API 字段不一致 | 设计与影响范围 |
| 后端测试失败 | 后端实现 |
| 前端联调失败 | API 契约或前端实现 |
| 布局溢出 | 前端样式与布局 |
| 回归失败 | 对应实现阶段 |

## 最小执行规则

即使是小需求，也至少执行：

1. 写清验收标准。
2. 说明影响范围。
3. 开发后运行相关测试或说明未运行原因。
4. 输出风险和下一步。
