# Definition of Done

一个需求只有满足本文件中的完成定义，才建议标记为 Done。

## 必须满足

- [ ] 需求目标已实现。
- [ ] 验收标准逐条确认并记录结果。
- [ ] 相关自动化测试已运行，或明确说明未运行原因。
- [ ] 主流程已手动验证，或明确说明阻塞点。
- [ ] API 变更已同步前后端。
- [ ] 没有违反后端三层架构。
- [ ] 没有违反项目 API 约定。
- [ ] 前端页面没有明显布局溢出或异常滚动。
- [ ] 最终总结包含修改内容、验证结果、风险和下一步。

## 后端完成标准

- [ ] schema/model/repository/service/router 分层清晰。
- [ ] router 不直接访问数据库。
- [ ] router 不包含业务逻辑。
- [ ] repository 不引用 HTTP 层。
- [ ] service 不处理 HTTP 细节。
- [ ] model 不使用 ForeignKey。
- [ ] 测试覆盖成功路径和关键失败路径。

## 前端完成标准

- [ ] API 封装与后端契约一致。
- [ ] 成功、失败、加载、空状态可用。
- [ ] 用户操作后有明确反馈。
- [ ] 不出现横向滚动。
- [ ] 固定头/底部栏与滚动区域布局正常。
- [ ] 视觉风格符合 `frontend/uni.scss` 设计系统。

## 测试完成标准

记录每个测试命令：

```md
- 命令：
- 结果：通过 / 失败 / 未运行
- 说明：
```

如果测试失败，不能标记 Done；应标记为 Blocked 或 In Progress，并说明失败原因。

## 可接受的未完成项

只有满足以下条件，未完成项才可以带入后续：

- 不影响本需求核心验收标准。
- 已明确记录风险。
- 已明确后续处理方式。
- 用户接受该风险。

## 最终回复模板

```md
## 完成情况
- ...

## 验收结果
- [x] ...
- [ ] ...（原因）

## 测试结果
- 命令：...
- 结果：...

## 风险
- ...

## 下一步
- ...
```
